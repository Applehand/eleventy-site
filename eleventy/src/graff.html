    How to read it

    Every node is a Schema.org entity with a stable @id; every edge is a Schema.org property linking two entities. Labels show the entity name and its primary type.

    • Organization: the business. Content points at it via publisher (articles, videos) or brand (products).
    • WebSite: the site container. Pages declare isPartOf → WebSite; it declares publisher → Organization.
    • WebPage nodes: one per template page. Each points at its subject via mainEntity and at its breadcrumb trail.
    • BreadcrumbList: the Home → template trail Google shows in results and agents use to locate a page in the site hierarchy.
    • Missing edges are usually schema.org domain rules, not omissions. A Product cannot declare isPartOf because it is not a CreativeWork.

            About these snippets
            
    
              Each template gets one self-contained snippet: a @graph holding the
              page's primary entity (Product, Article, Recipe…), a WebPage node, its
              BreadcrumbList, and the shared Organization and WebSite. One block serves every
              page built from that template. Fill the per-page tokens
              ({% raw %}{{PAGE_URL}}{% endraw %},
              {% raw %}{{PAGE_TITLE}}{% endraw %}) from the page itself and the
              site-wide tokens ({% raw %}{{ORGANIZATION_LOGO_URL}}{% endraw %})
              once.
            
    
            
      
              
    • 
                @id fragments: entities are named by URL plus
                fragment: site-wide ones on the origin
                (/#organization, /#website), page-scoped ones on the
                page URL (#webpage, #breadcrumb). Keep them exactly as
                generated: crawlers merge nodes with the same @id across pages, and
                this joins per-page markup into one site-wide graph.
              
      • 
              
    • 
                Duplicate Organization and WebSite blocks: every snippet
                repeats them so each page stands alone. Search engines de-duplicate them by
                @id.
              
      • 
              
    • 
                Nested objects: Offer, PostalAddress, Person, and ListItem
                shapes match what Google's documentation specifies for each feature.
              
      • 
              
    • 
                Breadcrumb hub levels: middle crumbs come from the example
                path (/gear/tent yields Home > Gear > page). Confirm each hub
                URL exists as a real page, or remove that level.
              
      • 
              
    • 
                References: values like
                {"@id": "…/#organization"} are pointers to another node in the
                graph, not text. Leave them unchanged.

          Why structured data matters for AI search
          
    
            Search engines and AI assistants increasingly read pages as data, not prose. JSON-LD
            structured data gives crawlers and agents machine-readable statements about what a
            page contains, instead of inferring it from layout and copy.
          
    
          
      
            
    • 
              Google rich results (product cards, recipe carousels, breadcrumbs,
              job listings) are driven directly by this markup and its required properties.
            
      • 
            
    • 
              AI answer engines and agents use schema markup for grounding and
              entity resolution: a connected graph with stable @ids states "this
              page is about this product, sold by this organization, on this site" without inference.
            
      • 
            
    • 
              Navigation signals like BreadcrumbList and
              mainEntity tell agents where a page sits in the site hierarchy and
              what its primary subject is.
            
      • 
          
    
        
    
        
    
          How this tool works
          
    
            Every blueprint starts from a deterministic skeleton: an Organization → WebSite →
            WebPage spine with breadcrumbs and stable @id fragments, typed by
            word-boundary keyword rules. AI never builds structure from scratch; it refines a
            graph that is already valid.
          
    
          
      
            
    • 
              schema.org registry: the full v30 vocabulary (3,003 terms: 843
              types, 1,529 properties, plus enumerations) compiled into a local database. Every
              type and property in the output is validated against it: domains, ranges, and
              subtype relations. Unknown terms are dropped, not published.
              
            
      • 
            
    • 
              Gemini ops router: when AI is enabled, Gemini works through
              function-calling operations (search_schema_types,
              list_subtypes, get_allowed_properties,
              check_feature_eligibility, propose_template_types,
              add_entity_node, add_entity_facts,
              add_relationship, complete_routing). Each call is
              validated server-side, so the model can only make changes the registry allows. It
              browses the type hierarchy with list_subtypes to pick the most
              specific type the input supports (a bookstore becomes BookStore, not
              just Organization), and a final enrichment turn extracts facts
              explicitly stated in the input (addresses, phone numbers, named people, hours)
              into the graph. Entities it creates must be linked into the graph or they are
              dropped, so the result is always fully connected.
            
      • 
            
    • 
              Google feature manifests: 25 rich-result rule sets sourced from
              Search Central documentation, with a citation per property. They drive the
              required and recommended fields in each snippet and the eligibility panel above.
            
      • 
            
    • 
              Graceful degradation: if AI is unavailable or its output fails
              validation, generation falls back to pure deterministic mapping. The result is
              never below a valid, registry-checked baseline.
