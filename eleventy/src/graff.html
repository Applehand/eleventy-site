    How to read it

    Every node is a Schema.org entity with a stable @id; every edge is a Schema.org property linking two entities. Labels show the entity name and its primary type.

    • Organization: the business. Content points at it via publisher (articles, videos) or brand (products).
    • WebSite: the site container. Pages declare isPartOf → WebSite; it declares publisher → Organization.
    • WebPage nodes: one per template page. Each points at its subject via mainEntity and at its breadcrumb trail.
    • BreadcrumbList: the Home → template trail Google shows in results and agents use to locate a page in the site hierarchy.
    • Missing edges are usually schema.org domain rules, not omissions. A Product cannot declare isPartOf because it is not a CreativeWork.

    Rich result opportunities
    
            
    
              What makes a page eligible?
              
    
                Google publishes rules for each rich result type in Search Central: a set of
                required properties the markup must carry, and
                recommended properties that strengthen the result. A page
                qualifies when its markup uses a matching Schema.org type and every required
                property is present. The cards below track exactly that:
                Qualifies now means the requirements are met,
                Needs values means the structure is complete but placeholder
                values remain, and Missing data means a required property needs
                real nested content GRAFF won't invent.
              
    
              
    
                Rich result eligibility is the baseline; there are more fields to fill than just the required ones. If all things are equal, the most
                complete, truthful entity graph wins the competition. Aim past "qualifies" and fill as many fields as completely as possible.
              
    
       
            
    
            
    
          
    
        
    

        
    
          
    
            
    JSON-LD snippets
    
          
    
            
            
          
    
          
    
            Paste each block into a <script type="application/ld+json"> tag on
            matching pages. Replace every {% raw %}{{TOKEN}}{% endraw %} with real
            values before publishing. Blocks are editable in place; copies and exports use the
            edited text.
          
    
          
    
            About these snippets
            
    
              Each template gets one self-contained snippet: a @graph holding the
              page's primary entity (Product, Article, Recipe…), a WebPage node, its
              BreadcrumbList, and the shared Organization and WebSite. One block serves every
              page built from that template. Fill the per-page tokens
              ({% raw %}{{PAGE_URL}}{% endraw %},
              {% raw %}{{PAGE_TITLE}}{% endraw %}) from the page itself and the
              site-wide tokens ({% raw %}{{ORGANIZATION_LOGO_URL}}{% endraw %})
              once.
            
    
            
      
              
    • 
                @id fragments: entities are named by URL plus
                fragment: site-wide ones on the origin
                (/#organization, /#website), page-scoped ones on the
                page URL (#webpage, #breadcrumb). Keep them exactly as
                generated: crawlers merge nodes with the same @id across pages, and
                this joins per-page markup into one site-wide graph.
              
      • 
              
    • 
                Duplicate Organization and WebSite blocks: every snippet
                repeats them so each page stands alone. Search engines de-duplicate them by
                @id.
              
      • 
              
    • 
                Nested objects: Offer, PostalAddress, Person, and ListItem
                shapes match what Google's documentation specifies for each feature.
              
      • 
              
    • 
                Breadcrumb hub levels: middle crumbs come from the example
                path (/gear/tent yields Home > Gear > page). Confirm each hub
                URL exists as a real page, or remove that level.
              
      • 
              
    • 
                References: values like
                {"@id": "…/#organization"} are pointers to another node in the
                graph, not text. Leave them unchanged.

          Why structured data matters for AI search
          
    
            Search engines and AI assistants increasingly read pages as data, not prose. JSON-LD
            structured data gives crawlers and agents machine-readable statements about what a
            page contains, instead of inferring it from layout and copy.
          
    
          
      
            
    • 
              Google rich results (product cards, recipe carousels, breadcrumbs,
              job listings) are driven directly by this markup and its required properties.
            
      • 
            
    • 
              AI answer engines and agents use schema markup for grounding and
              entity resolution: a connected graph with stable @ids states "this
              page is about this product, sold by this organization, on this site" without inference.
            
      • 
            
    • 
              Navigation signals like BreadcrumbList and
              mainEntity tell agents where a page sits in the site hierarchy and
              what its primary subject is.
            
      • 
          
    
        
    
        
    
          How this tool works
          
    
            Every blueprint starts from a deterministic skeleton: an Organization → WebSite →
            WebPage spine with breadcrumbs and stable @id fragments, typed by
            keyword rules over your template names and descriptions. AI never builds structure
            from scratch — it refines a graph that is already valid, and every change it
            proposes has to pass the same server-side checks the skeleton does.
          
    
          
      
            
    • 
              schema.org registry: the full v30 vocabulary (3,003 terms: 843
              types, 1,529 properties, plus enumerations) compiled into a local database. Every
              type and property in the output — whether picked by keyword rules or proposed by
              AI — is checked against its domains, ranges, and subtype relations before it
              enters the graph, so nothing outside the vocabulary can reach your snippets.
              Rejected proposals are reported back to the model so it can correct course, and
              claims your input can't support are listed as unsupported instead of published.
              
            
      • 
            
    • 
              Gemini ops router: with AI enabled, Gemini refines the draft
              through a set of server-validated operations rather than free-form output. It
              searches the vocabulary and browses subtypes to find the most specific type your
              input supports (a bookstore becomes BookStore, not just
              Organization), and when a retype would cost a Google rich result,
              the server tells it so it can keep both the precise type and the eligible one.
              It extracts facts you explicitly stated — phone numbers, emails, hours, prices,
              founding dates — creates Person entities for named people, and
              builds structured values like postal addresses from fixed, server-owned shapes.
              Everything it creates must be linked into the graph or it is dropped.
            
      • 
            
    • 
              Focused follow-up passes: after the main mapping, dedicated
              turns sweep for stated facts the router missed, fill Google-required properties
              your input provides values for, and suggest example values for the placeholder
              tokens in each snippet. If the graph ever fails validation, the model receives
              the exact findings and gets a bounded chance to repair its own work before any
              fallback.
            
      • 
            
    • 
              Google feature manifests: 25 rich-result rule sets sourced from
              Search Central documentation, with a citation per property. They drive the
              required and recommended fields in each snippet, the eligibility panel above,
              and the placeholder tokens seeded into the skeleton.
            
      • 
            
    • 
              Graceful degradation: if AI is off, unavailable, over quota, or
              its output can't be repaired, generation falls back to the deterministic
              skeleton. The result is never below a valid, registry-checked baseline — the AI
              can only add to it, not break it.
